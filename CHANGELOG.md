# Changelog — OpenSankey

Format basé sur [Keep a Changelog](https://keepachangelog.com/fr/1.1.0/).

## [Non publié] — Avril 2026

### Ajouts / Refonte UI du dialogue Persistence
- **Refonte des groupes d'options** (`PersistenceProcessDialogConfigs.tsx`) : les groupes `validation` + `completion` sont fusionnés en un seul `autocorrection` (« Autocomplétion / autocorrection »). Le groupe `packaging` (« Conditionnement ») est supprimé ; `mode_compressed` (« Fichier ZIP ») rejoint le groupe `content`. Le groupe `content` (« Contenu écrit ») ne contient plus que `keep_siblings` + `with_values` + `mode_compressed` côté JSON, donc n'apparaît plus en sortie Excel. Les attributs Excel-output spécifiques (`data_table_*`, `flux_matrix_*`) et les options communes `with_values` / `save_only_visible_elements` migrent vers `sheets` (« Onglets écrits »). `with_sheet_formating` passe de `presentation` vers `merge`.
- **Inversion sémantique « bloquer » → « créer »** : les attributs `error_on_new_nodes` / `error_on_new_flux` (cocher = bloquer) sont remplacés côté UI par `create_new_nodes` / `create_new_flux` (cocher = créer, défaut décoché). `PersistenceProcessDialog` transforme avant d'envoyer au backend (`error_on_new_* = !create_new_*`), donc `load_sankey` reçoit toujours les kwargs originaux. Wording cohérent avec les checkboxes « Propager les flux aux enfants/parents ».
- **Visibilité conditionnelle inter-format** : `AutoGeneratedOptions` reçoit deux nouveaux props `input_format` / `output_format` et injecte des clés synthétiques `_input_format` / `_output_format` dans `mergedOptions`. Les `visibilityConditions` peuvent ainsi brancher sur le format opposé via `optionProperty`. Utilisé pour : `save_only_visible_elements` visible uniquement quand input = blob (Sankey courant), `with_values` déplacé en `OUTPUT_ATTRIBUTES_CONFIG.json` (visible uniquement output = JSON).
- **`only_current_view` aussi pour input blob** : ajouté à `INPUT_ATTRIBUTES_CONFIG.blob`. Le dialogue propage la valeur cochée dans `toJSON_kwargs.only_current_view` quand il sérialise le Sankey courant.
- **`save_only_visible_elements` rendu seul, sans header** : l'attribut perd son `group`. `AutoGeneratedOptions` place les attributs UNGROUPED en tête (au lieu de la fin) → l'option apparaît seule au-dessus de la première section nommée.
- **Saut de ligne entre `Onglet(s) données` et `Onglet(s) TER ou TES`** : `breakBefore: true` ajouté à `activate_flux_matrix` (output Excel). Pluralisation cohérente FR/EN/ES/DE/IT (« Onglet(s) ... », « Sheet(s) ... », etc.) sur les 4 attributs (input + output × data + matrix).
- **Pas de Tabs quand une seule section est visible** : si seul `input_options` OU `output_options` (ou layout) a du contenu, le panel est rendu directement sous *Options* sans `Tabs`/`TabList`/`Tab` ni titre d'onglet redondant.
- **Termal : prise en charge du niveau WARNING** : `LogDisplay` et `Counter` ([PersistenceProcessDialogTerminal.tsx](opensankey/client/src/components/dialogs/PersistenceProcessDialogTerminal.tsx)) affichent les lignes `WARNING` en orange foncé (`#b7791f`), sous le filtre *Infos* (utilisé pour signaler les créations / propagations qui modifient la structure).
- **Dialogue *Créer l'onglet TER/TES*** : `output_options_visible_excel` étend l'exposition à `flux_matrix_with_data` et `flux_matrix_only_leaf_flux` en plus de `activate_flux_matrix`.
- **Renommages i18n** : « Convertisseur de fichiers » → « Édition de fichier » (5 langues, dans [PersistenceProcessDialogConfigs.tsx](opensankey/client/src/components/dialogs/PersistenceProcessDialogConfigs.tsx) et [traduction_app_elements.tsx](opensankey/client/src/traductions/traduction_app_elements.tsx)). « Réécrire les onglets SankeyExcelParser » → « Réécrire les onglets spécifiques OpenSankey ». « Formattage des onglets excel » → « Formattage des onglets ». Label « Flux feuilles » → « Uniquement Flux feuilles ».

### Backend / converter
- **Converter `launch_conversion` n'envoie plus le flag `process_options.autocorrect`** : le toggle dédié *Auto-correction* est retiré de l'UI ; les checkboxes `create_new_*` et `propagate_flux_*` déclenchent désormais le coloriage rouge directement via le chemin standard load+write (voir [SankeyExcelParser CHANGELOG](submodules/SankeyExcelParser/CHANGELOG.md)). Le wiring `process_options` ajouté précédemment dans `launch_conversion` (et le rename depuis `solver_options`) est nettoyé. `run_autocorrect_pass` reste utilisable depuis le CLI MFAProblem.

### Ajouts précédents
- **Case « Auto-correction » dans le dialogue Réconciliation / Convertisseur** ([issue #1221](https://gitlab.com/su-model/opensankey/-/issues/1221)) : nouvelle case dans `INPUT_ATTRIBUTES_CONFIG.excel` (groupe `validation`, défaut décochée) — visible dans l'onglet *Options d'entrée* du `PersistenceProcessDialog` quand le format d'entrée est Excel, à côté de `error_on_new_nodes` et `error_on_new_flux`. Quand cochée, le frontend retire `autocorrect` de `input_options` (pour ne pas polluer `load_sankey`) et l'envoie au backend dans un payload séparé `solver_options={ autocorrect: true }`. Le suffixe du fichier téléchargé devient `_corrected` au lieu de `reconciled`. Côté Flask SankeyApplication, le handler `/optimize/launch_optim` court-circuite la réconciliation et renvoie un fichier Excel corrigé avec les cellules ajoutées / les triggers surlignés en rouge (voir [SankeyApplication CHANGELOG](../../CHANGELOG.md) et [SankeyExcelParser CHANGELOG](submodules/SankeyExcelParser/CHANGELOG.md)). Traduit fr/en/es/de/it via les `labels` / `tooltips` de la `FormatAttributeConfig`.
- **`extra_export_menu_items` extensible — groupes nommés et tooltips** : le type `extra_export_menu_items` exposé par `Class_MenuConfig` ([MenuConfig.tsx](opensankey/client/src/types/MenuConfig.tsx)) accepte désormais une union discriminée par un champ `type`. Variante `'item'` (défaut, optionnelle pour rétro-compat) = entrée plate comme avant. Variante `'group'` = section titrée avec `children: Item[]`, rendue par [MenuTop.tsx](opensankey/client/src/components/topmenus/MenuTop.tsx) via `<MenuDivider />` + `<MenuGroup title>` au lieu d'un `MenuItem` plat. Chaque item (ou enfant de groupe) accepte aussi un `tooltip?: () => string` ; quand le tooltip retourne une chaîne non vide, `MenuTop` enveloppe le `MenuItem` dans un `<OSTooltip placement='right'>` + `<Box>` (la `Box` interpose une cible interactive pour que le tooltip continue de s'afficher même quand le `MenuItem` est `isDisabled`). Permet à OpenSankey+ d'enregistrer le sous-menu *Toutes les vues* avec un tooltip explicatif quand la licence est absente ou qu'il n'y a aucune vue.

- **Édition inline des labels — double-click, menu contextuel, raccourci typing** ([issue #688](https://gitlab.com/su-model/opensankey/-/issues/688)) : trois entrées d'accès au mini-éditeur de label (foreignObject + input HTML déjà présent mais inaccessible). (a) **Double-click** sur le texte d'un label de nœud, sur la valeur d'un flux, ou sur le label d'une zone de texte ouvre directement l'input pré-sélectionné — `attachDoubleClickEdit(textElement)` dans [DrawLabel.tsx](opensankey/client/src/Elements/DrawLabel.tsx) attache un `dblclick` qui appelle `setInputLabelVisible()`. Pour `LinkDrawValueLabel`, l'attachment est fait dans le `finalizeLabelCreation` override (la classe utilisait `enableEditing=false` jusqu'ici) et un `onInputChange` numérique parse la saisie (virgule → point, vide → null) pour mettre à jour `link.valueCurrent`. Pour les zones de texte, le comportement est hérité naturellement de `Class_NodeBase` puisque `Class_ContainerElement` étend `NodeBase`. (b) **Menu contextuel** : entrée `editName` réactivée dans `NODE_MENU_CONFIG` ([ContextNodeConfig.tsx](opensankey/client/src/components/dialogs/ContextNodeConfig.tsx), précédemment commentée) qui invoque `nodeActions.editName` → `setInputLabelVisible()`. (c) **Raccourci typing** : taper un caractère imprimable (sans Ctrl/Alt/Meta) quand un seul élément éditable est sélectionné ouvre l'éditeur pré-rempli avec le caractère. Implémenté dans [ApplicationData.tsx](opensankey/client/src/types/ApplicationData.tsx) au début de `handleKeyDown`, avec garde sur `evtOnDrawingArea`, sur la longueur de la sélection (1 nœud OU 1 zone de texte, et 0 lien), et `setInputLabelVisible(initialValue?: string)` étendu pour accepter et pré-poser une valeur. Documenté dans la modale d'accueil — deux nouvelles entrées dans le tableau des raccourcis du mode sélection ([traduction_rcc_shortcuts.tsx](opensankey/client/src/traductions/traduction_rcc_shortcuts.tsx), 5 langues). Effet de bord corrigé : `setInputLabelVisible` ne faisait que `document.getElementById(inputId)?.focus()` sur un foreignObject `display:none` — donc rien ne se passait. La méthode révèle désormais le foreignObject (`style('display', null)`), masque le `<text>`, et utilise la séquence `select() + setSelectionRange(end)` pour caret en fin (sinon le caractère pré-rempli reste sélectionné et la prochaine frappe l'écrase). Pour les flux, `LinkDrawValueLabel.setInputLabelInvisible` override déclenche `link.drawWithNodes()` au blur — la saisie était wrappée dans `bypass_redraws=true` pour ne pas re-rendre à chaque frappe, donc épaisseur source/target des nœuds restait obsolète sans ce redraw final.

- **Raccourcis *Créer l'Index* et *Créer TER/TES*** ([SankeyApplication #135](https://gitlab.com/su-model/sankeyapplication/-/issues/135)) : nouvelles entrées `CONVERTER_CONFIGS.create_index` et `CONVERTER_CONFIGS.create_ter_tes` ([PersistenceProcessDialogConfigs.tsx](opensankey/client/src/components/dialogs/PersistenceProcessDialogConfigs.tsx)) — variantes de `save_excel` qui pré-cochent les sheets pertinents pour leur cas d'usage. - **Options d'enregistrement Excel — préservation des onglets utilisateur** : nouvelle case `keep_other_sheets` (def. false) qui copie l'input Excel vers l'output avant écriture, préservant les onglets non-format (uniquement effectif si l'input est Excel). Combinée à `rewrite_format_sheets` (qui remplace `mode_write` côté UI), permet 4 combinaisons claires : conserver/écraser × réécrire/laisser intacts les onglets format SankeyExcelParser. La sémantique est portée nativement par le parser (voir [SankeyExcelParser CHANGELOG](submodules/SankeyExcelParser/CHANGELOG.md)).
- **Options d'enregistrement Excel — restriction aux nœuds feuilles** : deux nouvelles sous-options `data_table_only_leaf_flux` (sous `activate_data_table`) et `flux_matrix_only_leaf_flux` (sous `activate_flux_matrix`). Si cochées, restreignent l'export du DATA_SHEET et de la matrice IO/TER aux flux dont les deux extrémités sont des nœuds sans enfants dans la hiérarchie d'agrégation. Permet d'avoir « Onglet données | Inclure flux sans données | Flux feuilles » sur une seule ligne.
- **Verrou utilisateur sur le statut recyclage d'un flux** ([issue #711](https://gitlab.com/su-model/opensankey/-/issues/711)) : nouvel attribut booléen `shape_is_recycling_locked` sur les liens (default `false`), exposé via un petit cadenas (`FaLock`/`FaLockOpen`) placé à droite du bouton recyclage dans le menu Apparence des flux. Quand verrouillé, le calcul auto-sankey conserve la valeur `shape_is_recycling` définie par l'utilisateur au lieu de la recalculer depuis la détection de cycles. Côté algorithme ([NodePositioning](opensankey/client/src/Algorithms/NodePositioning.tsx)) : (a) les liens locked-true sont pré-amorcés dans `possible_recycling_links_ids` avant le DFS pour que la détection de cycles les considère déjà coupés et ne descende pas par eux pour calculer `position_u` ; (b) le post-mark de la branche d'auto-compute (regular branch) utilise désormais directement le set `possible_recycling_links_ids` comme source de vérité au lieu d'une comparaison `node_index >= target_index`. La comparaison d'index re-flaguait les arêtes restantes du cycle (qui se retrouvent backward après relaxation topologique des index une fois le cycle coupé), produisant deux flux marqués comme recyclage là où l'utilisateur n'en avait verrouillé qu'un seul. Sémantique « candidate » : un seul flux locked-true suffit à forcer les autres du même cycle à non-recyclage. Le skip_horizontal-branch et `applyComputedHorizontalIndexes` continuent d'utiliser l'index comparison (pas de cycle detection disponible) mais skippent les liens verrouillés. Limitation : locked-false n'a pas de support algorithmique dédié — si un cycle n'a aucun locked-true, le DFS choisit librement quelle arête couper.
- **Refonte UI du panneau Apparence des flux** : tous les boutons de forme (recyclage + cadenas + 4 orientations hh/vv/vh/hv) sont désormais sur une seule ligne via un grid `2fr 1fr 4fr` (le bouton recyclage est élargi). Les checkboxes *Structure* et *Courbe* + sélecteur de chemin bézier sont regroupées sur une seconde ligne en `menuconfigpanel_row_2cols`. L'icône recyclage utilise désormais `<FaRecycle />` de `react-icons/fa` (l'ancien SVG `logo_recycle` à viewBox 52×26 s'écrasait dans le carré 1rem² imposé par `normalizeIcon` et apparaissait illisible).
- **Poignées Bézier auto-réglées à 1 % sur passage en recyclage** : le setter `customShapeIsRecycling` ([Element.tsx](opensankey/client/src/Elements/Element.tsx)) sur transition `false → true` ramène désormais `shape_starting_tangeant` et `shape_ending_tangeant` à `0.01` (= 1 % en UI), au lieu de laisser le défaut `0.25` (= 25 %) qui produisait des points de contrôle Bézier trop éloignés des extrémités. Concerne le clic utilisateur sur le bouton Recyclage, le toggle automatique en self-loop (`source === target`) dans [Link.tsx](opensankey/client/src/Elements/Link.tsx), et le marquage automatique par `computeAutoSankey` quand un flux passe non-recyclage → recyclage. Le `fromJSON` écrit directement dans `attributes` sans passer par le setter — les valeurs sauvegardées par l'utilisateur sont donc préservées au chargement, et seuls les flux pour lesquels le statut recyclage change vraiment voient leurs poignées ré-initialisées.

### Corrections
- **« Préserver les colonnes additionnelles » déplacé d'Options d'entrée vers Options de sortie** ([issue #1220](https://gitlab.com/su-model/opensankey/-/issues/1220)) : la case `preserve_extra_columns` était exposée dans l'onglet *Options d'entrée Excel* (groupe `content`), ce qui suggérait à tort qu'elle ne concernait que la lecture. La décision est sémantiquement une décision d'écriture (réintroduire les colonnes utilisateur dans les feuilles régénérées), même si le stash s'arme côté lecture. Déplacée vers l'onglet *Options de sortie Excel* (groupe `merge`, à côté de *Conserver les autres onglets* et *Réécrire les onglets SankeyExcelParser*). Côté serveur, `views.py` propage désormais `output_options["preserve_extra_columns"]` vers `input_options` avant `load_sankey` pour que la phase de lecture arme bien le stash. Tooltip enrichi pour expliciter cette propagation. Bug latent racine corrigé côté SankeyExcelParser (voir [SankeyExcelParser CHANGELOG](submodules/SankeyExcelParser/CHANGELOG.md)).
- **Réconciliation in-place dans une vue ne reconstruit plus le diagramme entier** : quand l'utilisateur lançait *Ajuster et compléter le diagramme* (action `afmReconciliation` du menu contextuel ZDD, configuration converter `reconciliation_sankey` en blob→blob) depuis une vue OSP, l'ancienne implémentation envoyait au serveur le master + toutes les vues sérialisées via `app_data.toJSON()`, recevait un JSON réconcilié, puis l'écrasait via `app_data.fromJSON(JSON_data, {}, false)`. Le serveur MFA ne connaissant pas la notion de vues OSP, le résultat était un diagramme sans vues — le master était remplacé et `_views` perdues. Refonte ciblée :
  - **Envoi** : `generic_process` ajoute désormais `only_current_view: true` aux kwargs de `app_data.toJSON()` quand `input_format == 'blob' && output_format == 'blob' && drawing_area.id !== default_main_sankey_id`. Côté OSP, `_toJSON` retourne alors uniquement le JSON de la vue courante (id remappé sur `default_main_sankey_id` pour le serveur).
  - **Réception** : nouveau paramètre `view_only: boolean = false` ajouté à `retrieveJSONResults`. Quand actif, on ne passe plus par `app_data.fromJSON` (qui écraserait `_views`). À la place : `app_data.reset({ only_current_view: true })` purge et recrée le DA courant via le hook OSP existant (qui préserve master + autres vues), puis `DrawingAreaPersistence.fromJSON(app_data.drawing_area, JSON_data, {})` charge le JSON serveur directement dans le DA de la vue. Le post-processing styles/legend/colors et la branche `apply_layout_current_sankey` opèrent ensuite sur ce DA.
  - **Persistence du cache compressé** : nouvelle méthode publique `saveCurrentViewToCache()` sur `Class_ApplicationData` (no-op par défaut), appelée à la fin de `retrieveJSONResults` quand `view_only` est vrai. OpenSankey+ override la méthode pour recompresser `_views[current_view_id].json` via `DrawingAreaPersistenceOSP.toJSON` — sans cela, le résultat de la réconciliation aurait été perdu au prochain switch de vue ou à la sauvegarde JSON, le cache compressé conservant l'état pré-réconciliation.
  - Détection du cas via `app_data.drawing_area.id !== default_main_sankey_id` côté `handleFinish`, qui propage le booléen à `retrieveJSONResults`. En mode master ou en OS pure (pas de vues), tous les guards retombent dans le chemin historique inchangé. dans [Utils.tsx](opensankey/client/src/types/Utils.tsx), la division par `label_values.unit_factor` était gatée par `label_values.unit_visible && label_values.unit != ''`. Conséquence : si l'utilisateur définissait un facteur d'échelle (ex. 1000 pour passer de g à kg) mais sans cocher « Unité » ou sans nommer l'unité, le facteur n'était pas appliqué et la valeur s'affichait à l'échelle d'origine. Les gardes `unit_visible` et `unit != ''` sont retirées ; seuls `!is_percent` et `unit_factor > 1` subsistent. Le facteur agit désormais comme une échelle de display pure, dissociée de l'affichage du texte de l'unité.
- **Export PNG/PDF — labels rich text (foreignObject) rendus correctement** : l'ancien `pre_process_export_svg` dans [ApplicationData.tsx](opensankey/client/src/types/ApplicationData.tsx) remplaçait chaque `<foreignObject>` par un unique `<text>` SVG avec `textContent` brut — gras partiel détecté, mais tout le reste (italique, souligné, couleur par span, multi-ligne, wrap à `max-width`, alignement par paragraphe) était perdu. - **Refonte du mode de positionnement paramétrique — PR 3 (centralisation + containers récursifs + drag local + clustering de colonnes)** : troisième et dernière passe de la refonte ([issue #1210](https://gitlab.com/su-model/opensankey/-/issues/1210)), faisant suite aux PR 1 et PR 2. Aucun comportement utilisateur nouveau « visible » dans le cas nominal, objectif : rendre `Class_NodePositioning.recomputeParametricLayout` le **seul** chemin qui écrit `position_y` en mode parametric, appelé une fois par cycle de dessin depuis `Class_DrawingArea.drawElements`.
  - **`Node.applyPosition` parametric → pass-through** : la branche `parametric` de [Node.applyPosition](opensankey/client/src/Elements/Node.tsx) ne contient plus le walk `nodeAbove → y = nodeAbove.y + nodeAbove.h + dy`. La clause de sortie « nested container bypass » (nœud à la fois parent d'un container et enfant d'un autre → calcul paramétrique skippé) est supprimée. `applyPosition` ne fait plus que poser le `transform` SVG depuis `position_x`/`position_y` déjà calculés par le recompute centralisé. La branche `relative` (import/export) est inchangée.
  - **`recomputeParametricLayout(scope)` en trois phases** : nouvelle méthode publique sur [Class_NodePositioning](opensankey/client/src/Algorithms/NodePositioning.tsx), invoquée en tête de `drawElements` quand le style par défaut est en `parametric`. **Phase A (sizing bottom-up)** : pour chaque container top-level, descente récursive post-order, mise à jour de `shape_min_height` (= `totalStackHeight(children) + margin_top + margin_bottom`) et `shape_min_width` (= `max(child width) + margins`) sans lire le SVG (géométrie logique seulement, pas de `getBBox`) — élimine la race qui motivait le `setTimeout(0)` historique. **Phase B (top-level)** : groupe les nœuds visibles non-échange non-enfants-de-container par `position_u`, trie chaque colonne par `position_v` (tie-break `position_y`), et `stackNodesVertically` avec pour ancre le `position_y` courant du nœud de plus petit V. Les containers top-level participent comme des nœuds normaux avec leur hauteur déjà correcte. **Phase C (top-down)** : chaque container top-level est redescendu récursivement pour re-stacker ses enfants à l'ancre `container.y + margin_top`. Scopes supportés : `'all'`, `'column' { u }`, `'subtree' { node }`.
  - **Drag parametric explicite en fin de drag** : [NodeEventsHandler.handleMouseDragEnd](opensankey/client/src/Elements/NodeEventsHandler.tsx) en mode parametric fait désormais un settle en 5 étapes ordonnées : `inferPositionUFromX` (avec clustering, cf. infra), reset `position_v = -1` pour les nœuds non-v-locked (fix d'un bug préexistant où le reset ignorait le verrou), `computeParametrization(false)` qui re-trie par y et réassigne V (c'est là que le swap implicite sur croisement de voisin se produit — pas besoin de détection explicite), `backCalculateShapePositionDyFromY` qui recalcule chaque `shape_position_dy` depuis les y courants (clamp ≥ 0), puis `drawing_area.drawElements()` explicite qui déclenche `recomputeParametricLayout` (phases A/B/C) immédiatement pour un re-stacking complet post-drag, containers inclus.
  - **Limitation connue documentée** : dragger l'enfant de plus petit `position_v` d'un container le fait revenir à l'ancre `container.y + margin_top` au re-stack de phase C, parce que cette ancre est fixe. Dragger un enfant quelconque non-premier-V fonctionne normalement. Fix demanderait soit de dériver l'ancre de phase C depuis le premier enfant courant (en propageant à `container.y`), soit de déplacer le container lui-même — hors scope.
  - **`inferPositionUFromX` par clustering** (fix d'un bug de frontière de rounding) : l'ancienne implémentation faisait `Math.round(x / dx)` sur chaque nœud indépendamment. Deux nœuds visuellement alignés à 1-2 px près pouvaient tomber de part et d'autre de la frontière de rounding `x = (u + 0.5) * dx` et finir dans des colonnes différentes. Observé sur un vrai diagramme avec `dx = 200` et quatre nœuds à `x ≈ 1898.88 / 1900.18 / 1901.47` : deux côtés de 1900, deux `u` différents. La nouvelle implémentation trie les nœuds par `x`, les regroupe en clusters par proximité (un nœud rejoint le cluster courant si son x est à moins de `tolerance = max(10, dx * 0.05)` du max-x du cluster), puis calcule un `u` unique par cluster. Un cluster contenant un nœud `u_locked` hérite de son `u`. Les autres utilisent `round(mean_x / dx)`.
  - **Nettoyage des chemins obsolètes** : suppression de `Class_NodeElement.restackContainerChildren` (remplacé par phase C), `Class_NodeElement.restackAncestorContainers` (remplacé par phase A récursive), le `setTimeout(0)` dans [Class_NodeDimension.setContainerMode](opensankey/client/src/Elements/NodeDimension.tsx) (remplacé par un `drawing_area.drawElements()` explicite). `propagateContainerEnvelopeToAncestors` est conservé — encore utilisé par `eventMouseDrag` pour le drag en mode absolu. `setContainerModeQuiet` est conservé — encore utilisé par `UpdateFrom` lors des changements de vue. Ligne commentée obsolète retirée dans [Toolbar.tsx](opensankey/client/src/components/topmenus/Toolbar.tsx). Net : ~55 lignes de code mort retirées.

- **Assainissement du mode de positionnement paramétrique (PR 1 + PR 2)** : première passe de refonte du mode parametric, préparant l'unification complète du recompute (PR 3 à venir). Deux axes : (a) suppression des bugs qui empêchaient le mode de se rafraîchir correctement ; (b) unification de `shape_position_dy` comme seule source de vérité pour l'espacement vertical entre nœuds d'une même colonne (ou sous-colonne container). Aucun changement utilisateur visible dans le cas nominal ; comportements corrigés ci-dessous étaient jusqu'ici silencieusement buggés.

Corrigé pour donner la sémantique attendue « même container OU aucun container des deux côtés ».
  - **fix**: en mode parametric, un changement de valeur d'un data tag modifie `getShapeHeightToUse()` (hauteurs de nœuds) mais l'espacement entre nœuds ne se rafraîchissait pas. La méthode fait désormais un back-calcul : pour chaque colonne (groupée par `position_u`), tri par `position_y` courant puis calcul `shape_position_dy = y_i - (y_{i-1} + h_{i-1})` pour chaque nœud sauf le premier. Les valeurs négatives (overlap préexistant) sont clampées à 0 avec un `console.warn` comptant les nœuds concernés — le design PR 2+ interdit les dy négatifs.   - **unification des gaps**: tous les sites qui empilaient verticalement avec un `gap = 10` hardcodé sont migrés sur deux nouveaux helpers statiques `Class_NodePositioning.stackNodesVertically(nodes, anchor_y)` et `Class_NodePositioning.totalStackHeight(nodes)`. L'invariant canonique est `n_0.y = anchor_y ; n_{i+1}.y = n_i.y + n_i.height + n_{i+1}.shape_position_dy` (le dy du premier nœud est ignoré). Call sites migrés : `Class_NodeElement.restackContainerChildren` / `restackAncestorContainers` (param `gap` supprimé) ; bloc de stack initial de `Class_NodeDimension.setContainerMode` ; `computeEffectiveBlockHeight` / `rebalanceAncestorColumns` / `updateNodePositioning` / `disaggregate` dans [Hierarchies.tsx](opensankey/client/src/Algorithms/Hierarchies.tsx) (param `vertical_gap` supprimé sur les helpers). `updateAggregationExpansionPositioning` utilisait déjà `aggregateNode.shape_position_dy` — aucun changement.

À l'entrée en mode container ou à la désagrégation, l'espacement initial des enfants passe donc de 10 à 20 px. Pour restaurer un défaut à 10 px il faut modifier la valeur dans le style, pas la réintroduire en dur.
