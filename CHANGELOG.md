## [Non publié] — Avril 2026

### Modifications
- **Refonte du mode de positionnement paramétrique — PR 3 (centralisation + containers récursifs + drag local + clustering de colonnes)** : troisième et dernière passe de la refonte ([issue #1210](https://gitlab.com/su-model/opensankey/-/issues/1210)), faisant suite aux PR 1 et PR 2. Aucun comportement utilisateur nouveau « visible » dans le cas nominal, objectif : rendre `Class_NodePositioning.recomputeParametricLayout` le **seul** chemin qui écrit `position_y` en mode parametric, appelé une fois par cycle de dessin depuis `Class_DrawingArea.drawElements`.
  - **`Node.applyPosition` parametric → pass-through** : la branche `parametric` de [Node.applyPosition](opensankey/client/src/Elements/Node.tsx) ne contient plus le walk `nodeAbove → y = nodeAbove.y + nodeAbove.h + dy`. La clause de sortie « nested container bypass » (nœud à la fois parent d'un container et enfant d'un autre → calcul paramétrique skippé) est supprimée. `applyPosition` ne fait plus que poser le `transform` SVG depuis `position_x`/`position_y` déjà calculés par le recompute centralisé. La branche `relative` (import/export) est inchangée.
  - **`recomputeParametricLayout(scope)` en trois phases** : nouvelle méthode publique sur [Class_NodePositioning](opensankey/client/src/Algorithms/NodePositioning.tsx), invoquée en tête de `drawElements` quand le style par défaut est en `parametric`. **Phase A (sizing bottom-up)** : pour chaque container top-level, descente récursive post-order, mise à jour de `shape_min_height` (= `totalStackHeight(children) + margin_top + margin_bottom`) et `shape_min_width` (= `max(child width) + margins`) sans lire le SVG (géométrie logique seulement, pas de `getBBox`) — élimine la race qui motivait le `setTimeout(0)` historique. **Phase B (top-level)** : groupe les nœuds visibles non-échange non-enfants-de-container par `position_u`, trie chaque colonne par `position_v` (tie-break `position_y`), et `stackNodesVertically` avec pour ancre le `position_y` courant du nœud de plus petit V. Les containers top-level participent comme des nœuds normaux avec leur hauteur déjà correcte. **Phase C (top-down)** : chaque container top-level est redescendu récursivement pour re-stacker ses enfants à l'ancre `container.y + margin_top`. Scopes supportés : `'all'`, `'column' { u }`, `'subtree' { node }`.
  - **Drag parametric explicite en fin de drag** : [NodeEventsHandler.handleMouseDragEnd](opensankey/client/src/Elements/NodeEventsHandler.tsx) en mode parametric fait désormais un settle en 5 étapes ordonnées : `inferPositionUFromX` (avec clustering, cf. infra), reset `position_v = -1` pour les nœuds non-v-locked (fix d'un bug préexistant où le reset ignorait le verrou), `computeParametrization(false)` qui re-trie par y et réassigne V (c'est là que le swap implicite sur croisement de voisin se produit — pas besoin de détection explicite), `backCalculateShapePositionDyFromY` qui recalcule chaque `shape_position_dy` depuis les y courants (clamp ≥ 0), puis `drawing_area.drawElements()` explicite qui déclenche `recomputeParametricLayout` (phases A/B/C) immédiatement pour un re-stacking complet post-drag, containers inclus.
  - **Limitation connue documentée** : dragger l'enfant de plus petit `position_v` d'un container le fait revenir à l'ancre `container.y + margin_top` au re-stack de phase C, parce que cette ancre est fixe. Dragger un enfant quelconque non-premier-V fonctionne normalement. Fix demanderait soit de dériver l'ancre de phase C depuis le premier enfant courant (en propageant à `container.y`), soit de déplacer le container lui-même — hors scope.
  - **`inferPositionUFromX` par clustering** (fix d'un bug de frontière de rounding) : l'ancienne implémentation faisait `Math.round(x / dx)` sur chaque nœud indépendamment. Deux nœuds visuellement alignés à 1-2 px près pouvaient tomber de part et d'autre de la frontière de rounding `x = (u + 0.5) * dx` et finir dans des colonnes différentes. Observé sur un vrai diagramme avec `dx = 200` et quatre nœuds à `x ≈ 1898.88 / 1900.18 / 1901.47` : deux côtés de 1900, deux `u` différents. La nouvelle implémentation trie les nœuds par `x`, les regroupe en clusters par proximité (un nœud rejoint le cluster courant si son x est à moins de `tolerance = max(10, dx * 0.05)` du max-x du cluster), puis calcule un `u` unique par cluster. Un cluster contenant un nœud `u_locked` hérite de son `u`. Les autres utilisent `round(mean_x / dx)`.
  - **Nettoyage des chemins obsolètes** : suppression de `Class_NodeElement.restackContainerChildren` (remplacé par phase C), `Class_NodeElement.restackAncestorContainers` (remplacé par phase A récursive), le `setTimeout(0)` dans [Class_NodeDimension.setContainerMode](opensankey/client/src/Elements/NodeDimension.tsx) (remplacé par un `drawing_area.drawElements()` explicite). `propagateContainerEnvelopeToAncestors` est conservé — encore utilisé par `eventMouseDrag` pour le drag en mode absolu. `setContainerModeQuiet` est conservé — encore utilisé par `UpdateFrom` lors des changements de vue. Ligne commentée obsolète retirée dans [Toolbar.tsx](opensankey/client/src/components/topmenus/Toolbar.tsx). Net : ~55 lignes de code mort retirées.
  - Documentation technique complète dans [doc/sources/fr/dev/features/positionnement_parametrique.rst](../../doc/sources/fr/dev/features/positionnement_parametrique.rst) (miroir EN disponible), section PR 3.
- **Assainissement du mode de positionnement paramétrique (PR 1 + PR 2)** : première passe de refonte du mode parametric, préparant l'unification complète du recompute (PR 3 à venir). Deux axes : (a) suppression des bugs qui empêchaient le mode de se rafraîchir correctement ; (b) unification de `shape_position_dy` comme seule source de vérité pour l'espacement vertical entre nœuds d'une même colonne (ou sous-colonne container). Aucun changement utilisateur visible dans le cas nominal ; comportements corrigés ci-dessous étaient jusqu'ici silencieusement buggés.
  - **fix**: prédicat malformé dans la branche `parametric` de [Node.applyPosition](opensankey/client/src/Elements/Node.tsx) (parenthèses manquantes autour de `||` / `&&`, comparaison `nodeAbove != this` lisant un local `undefined` toujours vrai). Corrigé pour donner la sémantique attendue « même container OU aucun container des deux côtés ».
  - **fix**: en mode parametric, un changement de valeur d'un data tag modifie `getShapeHeightToUse()` (hauteurs de nœuds) mais l'espacement entre nœuds ne se rafraîchissait pas. [handleDataTagSelection](opensankey/client/src/components/topmenus/Toolbar.tsx) appelle désormais `drawing_area.nodePositioning.computeParametrization(false)` à la fin quand le style par défaut est en `parametric`, avant le redraw.
  - **fix**: la bascule absolu → paramétrique via [setParametricMode](opensankey/client/src/types/DrawingArea.tsx) provoquait un saut visuel systématique car `computeParametrization` re-empilait en ignorant les positions absolues. La méthode fait désormais un back-calcul : pour chaque colonne (groupée par `position_u`), tri par `position_y` courant puis calcul `shape_position_dy = y_i - (y_{i-1} + h_{i-1})` pour chaque nœud sauf le premier. Les valeurs négatives (overlap préexistant) sont clampées à 0 avec un `console.warn` comptant les nœuds concernés — le design PR 2+ interdit les dy négatifs. Sur un diagramme sans overlap, la bascule est visuellement stable.
  - **fix**: `finalizeOperation` dans [Hierarchies.tsx](opensankey/client/src/Algorithms/Hierarchies.tsx) purgeait `position_dy` sur chaque nouveau nœud juste après que le helper de positionnement l'ait posé. Toute personnalisation locale du dy était perdue à chaque désagrégation. Le `delete_attribute('position_dy')` est supprimé.
  - **découplage u/x**: `computeParametrization` re-dérivait inconditionnellement `position_u = round(position_x / dx)` en tête, créant un couplage bidirectionnel `u ↔ x` (un container qui modifie `position_x` via son enveloppe faisait dériver `position_u` au recompute suivant, qui recalculait alors `position_x`, etc.). Le calcul est extrait dans une nouvelle méthode publique `Class_NodePositioning.inferPositionUFromX()` que les callers appellent explicitement quand ils viennent de modifier une position absolue : `setParametricMode` (bascule abs→param), `NodeEventsHandler.handleMouseDragEnd` (fin de drag en mode absolu), `Hierarchies` (création de nœud lors d'une désagrégation latérale). `computeParametrization` ne touche plus à `position_u` sauf quand `use_horizontal_index=true` (analyse topologique explicite).
  - **unification des gaps**: tous les sites qui empilaient verticalement avec un `gap = 10` hardcodé sont migrés sur deux nouveaux helpers statiques `Class_NodePositioning.stackNodesVertically(nodes, anchor_y)` et `Class_NodePositioning.totalStackHeight(nodes)`. L'invariant canonique est `n_0.y = anchor_y ; n_{i+1}.y = n_i.y + n_i.height + n_{i+1}.shape_position_dy` (le dy du premier nœud est ignoré). Call sites migrés : `Class_NodeElement.restackContainerChildren` / `restackAncestorContainers` (param `gap` supprimé) ; bloc de stack initial de `Class_NodeDimension.setContainerMode` ; `computeEffectiveBlockHeight` / `rebalanceAncestorColumns` / `updateNodePositioning` / `disaggregate` dans [Hierarchies.tsx](opensankey/client/src/Algorithms/Hierarchies.tsx) (param `vertical_gap` supprimé sur les helpers). `updateAggregationExpansionPositioning` utilisait déjà `aggregateNode.shape_position_dy` — aucun changement.
  - **conséquence visuelle**: le `shape_position_dy` par défaut est `20` dans `elementStyleConfigs` ([ElementStyle.tsx](opensankey/client/src/Elements/ElementStyle.tsx)), contre le `10` en dur précédent. À l'entrée en mode container ou à la désagrégation, l'espacement initial des enfants passe donc de 10 à 20 px. Pour restaurer un défaut à 10 px il faut modifier la valeur dans le style, pas la réintroduire en dur.
  - **ce qui reste**: la clause « nested container bypass » dans `Node.applyPosition` (mode parametric) est toujours en place — un nœud à la fois parent d'un container et enfant d'un autre container skippe son calcul paramétrique. Ce sera traité par la PR 3 qui centralisera le recompute dans une nouvelle `recomputeParametricLayout(scope)` et transformera `Node.applyPosition` en simple pose du transform SVG.

Ajouts côté steps : chaque bouton du bandeau top (nouveau, ouvrir, enregistrer, exporter, mise en page) pris individuellement, chaque groupe de la toolbar bas (mode souris édition/sélection/pinceau, mode position paramétrique/absolu, ajustement H-V-plein écran, bouton d'aide `?`), le drawer de filtres latéral gauche (conditionnellement, uniquement si `hasVisibleFilters()` renvoie vrai et que le bouton `#buttonOpenFilterDrawer` existe dans le DOM au moment du `setSteps()`), puis la séquence complète du panneau de configuration droit (drawer, boutons d'onglets data/style, zones de contenu). Les transitions d'état de l'UI (ouverture de drawers, bascule d'onglets, activation d'éléments configurables) sont pilotées via quatre helpers locaux — `openConfigDrawer` / `closeConfigDrawer` (via `ref_menu_opened`), `switchConfigTab` (via `type_menu_configuration_selected` + `ref_to_menu_config_updater`), `ensureElementSelected` (via `elements_configurable_selected` + `toggleElementInConfigEdition`), `setFilterDrawer` (via `ref_close_filter_drawer`) — appelés depuis les callbacks `action` / `actionAfter` des steps. Les panneaux du menu config sont révélés automatiquement en sélectionnant les éléments configurables pertinents (`data`→`node`+`flow`, `style`→`DA`+`element`) juste avant le step qui pointe sur `.config_box`, pour que l'utilisateur voie une UI concrète et non un panneau vide. Auto-création de contenu démo (`ensureDemoContent` sur `action` du premier step `#g_drawing`) : si `nodes_list.length === 0`, création de 2 nœuds + 1 flux via `sankey.addNewDefaultLink()`, valeur 100 sur le lien via `link.valueCurrent = 100`, et trois tag groups démo via `addNodeTagGroup` / `addFluxTagGroup` / `addDataTagGroup` (un tag par défaut chacun, libellés traduisibles `guide.demo_{node|flux|data}_tagg_name`). Guards : `ensureDemoContent` ne recrée rien si l'utilisateur revient sur le premier step (check sur `demoRefs.created`) et n'écrase jamais un diagramme existant (check sur `nodes_list.length !== 0`). Classes CSS ajoutées sur les points d'ancrage qui n'en avaient pas : `menutop_button_new` / `menutop_button_open` / `menutop_button_save` / `menutop_button_export` / `menutop_button_mep` sur les `MenuButton` / `Button` de [MenuTop.tsx](opensankey/client/src/components/topmenus/MenuTop.tsx) ; `toolbar_bottom_mouse_mode` / `toolbar_bottom_position_mode` / `toolbar_bottom_stretch` sur les `ButtonGroup` de [MenuBottom.tsx](opensankey/client/src/components/topmenus/MenuBottom.tsx) et `toolbar_bottom_help` sur le bouton `?` ; `button_type_config_data` / `button_type_config_style` sur les `Button` d'onglet et `config_box` sur le `Box` de contenu de [SankeyMenus.tsx](opensankey/client/src/components/topmenus/SankeyMenus.tsx). - **Verrouillage de l'axe sur le drag de nœud (Shift + glisser)** : maintenir la touche **Shift** pendant un drag de nœud contraint le déplacement à un seul axe (horizontal U ou vertical V), comme dans SankeyMatic. L'axe est figé après environ 4 px de mouvement cumulé, sur le composant dominant (``|acc_dx| >= |acc_dy|`` → axe X, sinon axe Y) ; en deçà du seuil les premiers pixels sont suppressés pour éviter qu'un léger jitter ne fuie en diagonale. Relâcher Shift en cours de drag libère immédiatement le verrou ; ré-appuyer réinitialise l'accumulateur et laisse re-choisir un axe à partir du mouvement suivant. Trois nouveaux champs privés `_shift_lock_axis` / `_shift_acc_dx` / `_shift_acc_dy` ajoutés à [NodeEventsHandler](opensankey/client/src/Elements/NodeEventsHandler.tsx) ; la logique vit en tête de `handleMouseDrag`, avant toute décision de déplacement, et mute `event.dx` / `event.dy` via `Object.defineProperty` (les propriétés du `DragEvent` d3 sont `configurable: true` mais `writable: false` — une assignation directe lève `TypeError: "dx" is read-only` en mode strict). La mutation étant en place sur la même référence, le verrou se propage automatiquement au mode magnétique (`moveMagneticNode` lit `event.dx/dy` avant le snap grille), à la propagation conteneur (`Class_NodeElement.eventMouseDrag` lit l'event après `super.eventMouseDrag` pour déplacer les enfants englobés) et aux drags multi-sélection. Les court-circuits `if (event.sourceEvent.shiftKey) return` placés auparavant en tête de `handleMouseDragStart` et `handleMouseDragEnd` ont été supprimés — ils cassaient silencieusement l'undo et l'auto-réorganisation E/S pour tout shift+drag. Une nouvelle entrée « Shift + Drag (noeuds) » a été ajoutée au tableau du mode sélection de la [modale d'accueil](opensankey/client/src/components/welcome/ModalWelcome.tsx), avec traductions FR/EN `Menu.rcc_shift_dn_bold` / `Menu.rcc_shift_dn` dans [traduction_rcc_shortcuts.tsx](opensankey/client/src/traductions/traduction_rcc_shortcuts.tsx).
- **Mode d'affichage englobant (container) sur les dimensions d'agrégation** : troisième mode d'affichage pour une dimension parent/enfant, en plus de l'agrégation et de la désagrégation classiques. Le parent est rendu en même temps que ses enfants, sous forme d'une enveloppe rectangulaire pointillée qui englobe visuellement les enfants, avec deux variantes de répartition des flux :
  - `in_children_out_parent` : flux entrants sur les enfants, flux sortants depuis le parent englobant ;
  - `in_parent_out_children` : flux entrants sur le parent, flux sortants depuis chaque enfant.
  Les flux internes au groupe (enfant→enfant de la même dimension) restent visibles dans les deux modes. Nouveau type `Type_ContainerMode` et champ privé `_container_mode` ajoutés à [Class_NodeDimension](opensankey/client/src/Elements/NodeDimension.tsx) avec `setContainerMode`, `unsetContainerMode`, `setContainerModeQuiet` (sync sans side effect pour `updateFrom`), et reset automatique des flags `force_show_*` quand le mode est activé. Visibilité asymétrique des liens implémentée via le nouveau getter `is_allowed_by_container_modes` sur [Class_LinkElement](opensankey/client/src/Elements/Link.tsx), intégré à `is_visible`. Rendu du parent englobant par une nouvelle méthode `applyContainerEnvelopeIfNeeded()` sur [Class_NodeElement](opensankey/client/src/Elements/Node.tsx), appelée au début de `_draw()`, récursive inward (les enveloppes inner sont mises à jour avant l'outer, pour supporter les containers imbriqués sans dépendre de l'ordre de draw). Nouveaux utilitaires partagés `_computeEnvelopeBBox` / `_applyEnvelopeBBox` remontés dans [Class_NodeBase](opensankey/client/src/Elements/NodeBase.tsx) — `_computeEnvelopeBBox` utilise uniquement la géométrie logique (position + `getShapeWidthToUse/HeightToUse`), pas de lecture SVG, ce qui garantit des résultats déterministes même avant que les updates soient flushés au DOM. [Class_ContainerElement](opensankey/client/src/Elements/TextZone.tsx) refactorisé pour déléguer à ces utilitaires. Drag du parent englobant : propagation `dx/dy` aux enfants contenus, en sautant ceux déjà dans la sélection pour éviter le double déplacement, via override de `eventMouseDrag` dans `Class_NodeElement`. Drag d'un nœud (feuille ou intermédiaire) : nouvelle méthode `propagateContainerEnvelopeToAncestors()` qui remonte la chaîne des containers (via `dimensions_as_child.container_mode`) et met à jour chaque ancêtre. Snapshot / restore de la géométrie pré-container (`saveGeometryForContainerMode` / `restoreGeometryAfterContainerMode` sur `Class_NodeElement`) pour que `unsetContainerMode` puisse restituer la position et la taille du parent avant activation. Nouveau style `NodeContainerStyle` enregistré dans [ElementStyle.tsx](opensankey/client/src/Elements/ElementStyle.tsx) (ajouté à `base_styles`) avec rectangle coins arrondis, bordure pointillée 2px, pas de remplissage, label centré au-dessus en gras, marges `shape_margin_left/right=5`, `shape_margin_top/bottom=10`. Style automatiquement appliqué via `addStyle` à l'activation du mode et retiré à la désactivation (protégé contre la suppression si une autre dimension du même parent reste en mode container). Les clics sur l'intérieur du container traversent via `pointer-events="visibleStroke"` appliqué dans [NodeDrawShape](opensankey/client/src/Elements/NodeDrawShape.tsx), ne laissant que la bordure cliquable — les petits nœuds enfants restent accessibles au clic droit. Persistance : `container_mode` sérialisé dans `NodeDimensionsManager.toJSON` et rejoué via `setContainerMode(mode, true)` dans `fromJSON` (fromJSON-aware : pas de stacking, pas de redraw interne). Sync lors du changement de vue (`updateFrom`) via `setContainerModeQuiet` dans le bloc `tagLevel`, aligné avec le sync existant de `force_show_children`. Entrées de menu contextuel « Englober (entrées → enfants, sorties ← parent) » et « Englober (entrées → parent, sorties ← enfants) » ajoutées dans les deux sous-menus dynamiques de « Navigation hiérarchie » ([SankeyMenuContext.tsx](opensankey/client/src/components/dialogs/SankeyMenuContext.tsx)), en cohabitation avec les entrées agrégation/désagrégation existantes, avec action de sortie « Quitter mode englobant » conditionnée à la présence d'un mode actif sur la dimension.
